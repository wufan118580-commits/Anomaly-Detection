# PostgreSQL TOAST 存储效率实验报告

## 一、实验背景与目的

### 1.1 实验目标
对比 PostgreSQL 数据库中两种数据存储方案的**空间效率**：
- **方案A (传统列式)**：每列独立字段存储，保持 Excel 原始列结构
- **方案B (TOAST列表式)**：将数值列合并为 `DOUBLE PRECISION[]` 数组存储，触发 PostgreSQL TOAST 机制

### 1.2 TOAST 机制简介
```
TOAST = The Oversized-Attribute Storage Technique
- 触发条件：单行某个变长字段 > 2KB（页面大小的 1/4）
- 工作原理：将大字段从主表移出，主表只保留指针（24 bytes）
- 目的：优化主表空间利用率，减少 I/O 开销
```

### 1.3 测试数据来源
| 项目 | 详情 |
|------|------|
| **文件** | DATA_TRACE.xlsx |
| **原始大小** | 87,407 bytes (85.36 KB) |
| **原始形状** | 446 行 × 24 列 |
| **时间戳列** | HH:MM:SS（精确到秒） |
| **数值列** | Ticks, L2AgentSn, BoardSn, ucChipNo, coreId, frmNo, slotNo, plmn, tti, msgCnt, gNodeBId, rsCellId, isLinkNormal, longtitude, latitude, elevation, speed, speedX, speedY, speedZ（共20个） |

## 二、数据库表结构设计

### 2.1 方案A：传统列式存储 (`storage_method_a`)

```sql
CREATE TABLE storage_method_a (
    id           SERIAL PRIMARY KEY,
    timestamp    TIMESTAMP NOT NULL,        -- 时间戳列
    rsCellId     DOUBLE PRECISION,          -- 数值列1
    uuid         TEXT,                       -- 文本列
    coreId       DOUBLE PRECISION,          -- 数值列2
    ucChipNo     DOUBLE PRECISION,          -- ...
    msgCnt       DOUBLE PRECISION,
    speedZ       DOUBLE PRECISION,
    latitude     DOUBLE PRECISION,
    plmn         DOUBLE PRECISION,
    BoardSn      DOUBLE PRECISION,
    gNodeBId     DOUBLE PRECISION,
    L2AgentSn    DOUBLE PRECISION,
    isLinkNormal DOUBLE PRECISION,
    speed        DOUBLE PRECISION,
    endian       TEXT,
    elevation    DOUBLE PRECISION,
    longtitude   DOUBLE PRECISION,
    Ticks        DOUBLE PRECISION,
    slotNo       DOUBLE PRECISION,
    tti          DOUBLE PRECISION,
    frmNo        DOUBLE PRECISION,
    speedY       DOUBLE PRECISION,
    speedX       DOUBLE PRECISION
);
-- 总计: 24 列 (id + timestamp + 22 个数据列)
-- 主键索引: storage_method_a_pkey (btree on id)
```

### 2.2 方案B：TOAST 列表式存储 (`storage_method_b`)

```sql
CREATE TABLE storage_method_b (
    id        SERIAL PRIMARY KEY,
    timestamp TIMESTAMP NOT NULL,            -- 时间戳列
    values    DOUBLE PRECISION[]             -- 所有数值合并为数组
);
-- 总计: 3 列 (id + timestamp + values[])
-- 主键索引: storage_method_b_pkey (btree on id)
-- values 字段类型为 PostgreSQL 数组，支持自动 TOAST
```

### 2.3 表结构对比

| 对比项 | 方案A (传统列式) | 方案B (TOAST列表式) |
|--------|-----------------|---------------------|
| **总列数** | 24 列 | 3 列 |
| **数据列数** | 22 列 | 1 列 (数组) |
| **主键** | SERIAL PRIMARY KEY | SERIAL PRIMARY键 |
| **时间戳类型** | TIMESTAMP NOT NULL | TIMESTAMP NOT NULL |
| **数值存储** | 每列独立 `DOUBLE PRECISION` | 统一 `DOUBLE PRECISION[]` |
| **TOAST 触发** | 不涉及 | 数组 > 2KB 时自动触发 |

## 五、结果分析

### 5.1 为什么数据库比 Excel 大？

| 原因 | 影响 | 说明 |
|------|------|------|
| **行头开销 (Tuple Header)** | ~23 bytes/行 | 每行固定开销，包括事务ID等 |
| **Null Bitmap** | ~3-6 bytes/行 | 记录哪些列为空 |
| **页面碎片** | ~5-10% | 最小单位 8KB 页面，尾部浪费 |
| **主键索引** | ~32 KB | B-tree 索引额外占用 |
| **数据对齐** | 变化 | 按 8 字节对齐，可能有填充 |
| **无压缩** | - | 默认不压缩数据 |

**计算示例 (446行)：**
```
行头: 23 bytes × 446 = 10,258 bytes (~10 KB)
Null bitmap: 3 bytes × 446 = 1,338 bytes (~1.3 KB)  
索引: ~32 KB
页面碎片: ~15 KB
合计额外: ~58 KB → 解释了 Excel(85KB) vs DB(168KB) 的差距
```

### 5.2 为什么方案B 反而更大？（关键发现！）

**核心原因：PostgreSQL 数组类型的元数据开销**

```
┌─────────────────────────────────────────────────────────────┐
│  方案A: 每行数据连续存储                                      │
│  ┌───────┬──────┬────┬──────┬──────┬──────┐                 │
│  │header │  id  │ ts │ col1 │ ...  │ col22│                 │
│  └───────┴──────┴────┴──────┴──────┴──────┘                 │
│  23b     4b    8b   8b           8b                         │
│  总计: ~199 bytes/行                                        │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│  方案B: 数组有额外的元数据开销                                │
│  ┌───────┬──────┬────┬──────────┬────────────────────────┐  │
│  │header │  id  │ ts │array_hdr │ varlena[1]...varlena[n]│  │
│  └───────┴──────┴────┴──────────┴────────────────────────┘  │
│  23b     4b    8b   24b        1b...1b                      │
│  总计: ~239 bytes/行                                        │
│  ↑ 数组头(24b) + 每元素 varlena(1b×20=20b) = +44 bytes     │
└─────────────────────────────────────────────────────────────┘
```

### 5.3 每行详细计算对比

| 组成部分 | 方案A | 方案B | 说明 |
|---------|-------|-------|------|
| Tuple Header | 23 bytes | 23 bytes | 固定不变 |
| ID (int4) | 4 bytes | 4 bytes | 自增主键 |
| Timestamp | 8 bytes | 8 bytes | 时间戳 |
| Null Bitmap | 3 bytes (22列→ceil(23/8)) | 1 byte (2列→ceil(3/8)) | **方案B 省 2 bytes** |
| 数据内容 | 160 bytes (20×8) | 160 bytes (20×8) | 相同 |
| **Array 头部** | - | **24 bytes** | **方案B 多 24 bytes** |
| **Varlena 头部** | - | **20 bytes (20×1)** | **方案B 多 20 bytes** |
| **总计** | **~198 bytes/行** | **~240 bytes/行** | **方案B 多 ~42 bytes/行** |

### 5.4 446 行时为什么方案B 略小？

```
特殊原因: 446 行时方案B 有 2 个 TEXT 列 (uuid, endian) 被排除
         这些列在方案A 中占用了额外空间
         
当使用纯数值测试 (scale_experiment.py):
  - 排除了文本列的影响
  - 结果显示方案B 在所有数据量下都更大
```

## 六、TOAST 是否真正工作？

### 6.1 当前实验中的 TOAST 使用情况

```sql
-- 查询 TOAST 表实际使用情况
SELECT 
    c.relname AS table_name,
    pg_size_pretty(pg_table_size(c.oid)) AS table_size,
    CASE WHEN c.reltoastrelid != 0 THEN 
        pg_size_pretty(pg_total_relation_size(c.reltoastrelid))
    ELSE '未使用' END AS toast_size
FROM pg_class c
WHERE c.relname IN ('storage_method_a', 'storage_method_b');

-- 结果:
-- storage_method_a: 144 KB, TOAST: 8 KB (预分配)
-- storage_method_b: 136 KB, TOAST: 8 KB (预分配)
```

**结论：两种方案的 TOAST 表都只有 8 KB（系统预分配），没有真正触发 TOAST 存储！**

### 6.2 未触发 TOAST 的原因

| 参数 | 当前值 | TOAST 阈值 | 是否触发 |
|------|--------|-----------|---------|
| 数组长度 | 20 个元素 | >256 个元素 | ❌ 否 |
| 数组大小 | 160 bytes | >2048 bytes | ❌ 否 |
| 单行总大小 | ~240 bytes | >8192 bytes | ❌ 否 |

```
当前数组大小: 20 × float8 = 160 bytes
TOAST 阈值:   2048 bytes
比例:          160 / 2048 = 7.8%

需要数组达到约 256 个 float 才能超过阈值!
```

### 6.3 TOAST 何时能真正发挥作用？

| 场景 | 条件 | 效果 |
|------|------|------|
| **大数组** | 元素数 > 256 | 移到 TOAST 表，主表只存指针 |
| **大文本** | 单字段 > 2KB | 自动压缩 + 外置存储 |
| **重复数据** | 启用压缩 | 压缩率可达 50-90% |
| **JSONB** | 结构化大文档 | 压缩效果显著 |

## 七、实验结论

### 7.1 核心结论

```
┌─────────────────────────────────────────────────────────────┐
│                    实验核心结论                               │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ❌ 对于小数据集 (20列 × <1000行)，列合并存储反而更大!       │
│                                                             │
│  ✓ 节省主要来自"列头开销"，而非 TOAST 压缩                   │
│                                                             │
│  ✓ TOAST 只在大字段 (>2KB) 时才真正工作                      │
│                                                             │
│  ✓ 数据库比 Excel 大是正常的（有元数据和索引开销）            │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 7.2 适用场景总结

| 场景 | 推荐 | 原因 |
|------|------|------|
| 少列 (<10列)、少行 (<1000) | **方案A (传统列式)** | 数组元数据开销更大 |
| 多列 (>50列)、大数据 | **方案B (TOAST列表)** | 可能触发 TOAST，节省主表空间 |
| 需要灵活查询某列 | **方案A (传统列式)** | 可以直接索引单列 |
| 批量读写整行数据 | **方案B (TOAST列表)** | 减少网络传输 |
| 包含大文本/BLOB | **方案B (TOAST列表)** | 自动利用 TOAST 压缩 |

### 7.3 后续建议

如果想验证 TOAST 的真实效果，建议：

1. **增大数组规模**: 测试 500+ 元素的数组
2. **启用压缩**: 设置 `ALTER TABLE t SET (toast_tuple_target = 2000)`
3. **添加重复数据**: 让数组包含可压缩的重复模式
4. **测量 I/O 性能**: 对比查询速度而非仅看存储大小

## 八、TOAST 触发实验 (500列)

### 8.1 实验设计

为了真正触发 PostgreSQL 的 TOAST 机制，设计了一个新的实验：

- **数值列数**：500 列（原始为 20 列）
- **每行数组大小**：500 × 8 = **4,000 bytes (3.9 KB)**
- **TOAST 阈值**：~2,048 bytes (页面的 1/4)
- **触发条件**：✅ 数组大小 > TOAST 阈值 → **将触发 TOAST！**

### 8.2 实验配置

| 配置项 | 值 |
|--------|---|
| **数值列数** | 500 |
| **每行数组大小** | 4,000 bytes (3.9 KB) |
| **页面大小 (block_size)** | 8,192 bytes |
| **TOAST 阈值** | ~2,048 bytes |
| **是否触发 TOAST** | ✅ **是** |

### 8.3 实验表结构

```sql
-- 方案A: 传统列式 (500个独立字段)
CREATE TABLE toast_a (
    id        SERIAL PRIMARY KEY,
    timestamp TIMESTAMP NOT NULL,
    col_0     DOUBLE PRECISION,      -- 第1列
    col_1     DOUBLE PRECISION,      -- 第2列
    ...                             -- ...
    col_499   DOUBLE PRECISION       -- 第500列
);
-- 总计: 502 列 (id + timestamp + 500个数据列)

-- 方案B: TOAST列表式 (1个数组)
CREATE TABLE toast_b (
    id        SERIAL PRIMARY KEY,
    timestamp TIMESTAMP NOT NULL,
    values    DOUBLE PRECISION[]    -- 所有500个数值存为数组
);
-- 总计: 3 列 (id + timestamp + values[])
```

### 8.4 实验结果

#### 存储空间对比

| 行数 | 方案A (传统) | 方案B (TOAST) | 差异 | 差异率 |
|------|-------------|---------------|------|-------|
| **100** | 434,176 bytes (**424.0 KB**) | 598,016 bytes (**584.0 KB**) | -163,840 bytes | **-37.74%** ❌ |
| **1,000** | 4,120,576 bytes (**4,024.0 KB**) | 5,677,056 bytes (**5,544.0 KB**) | -1,556,480 bytes | **-37.77%** ❌ |

> 注：负值表示方案B比方案A更大

#### 含索引总大小

| 行数 | 方案A 总计 | 方案B 总计 | 说明 |
|------|-----------|-----------|------|
| 100 | 450,560 bytes (**440.0 KB**) | 614,400 bytes (**600.0 KB**) | +36.6% |
| 1,000 | 4,161,536 bytes (**4,064.0 KB**) | 5,718,016 bytes (**5,584.0 KB**) | +37.3% |

### 8.5 TOAST 触发验证

```
TOAST 状态分析:

  ID |   数组长度 | 每行大小(bytes) | 触发TOAST?
-----+-----------+----------------+-----------
   1 |        500 |           4020 |         YES ✓
   2 |        500 |           4020 |         YES ✓
   3 |        500 |           4020 |         YES ✓
   4 |        500 |           4020 |         YES ✓
   5 |        500 |           4020 |         YES ✓

结论: 每行 4020 bytes > 2048 bytes 阈值 → 成功触发 TOAST!
```

### 8.6 结果分析：为什么触发了 TOAST 反而更大？

#### 存储结构对比

```
┌─────────────────────────────────────────────────────────────┐
│  方案A: 传统列式存储                                         │
│  ┌───────────────────────────────────────────────────────┐ │
│  │ 主表 row: [header][ts][col1][col2]...[col500]          │ │
│  │            23b   8b   8b   8b       8b                 │ │
│  │                                                       │ │
│  │ 数据: 500 × 8 = 4,000 bytes                            │ │
│  │ 元数据: header(23) + ts(8) = 31 bytes                  │ │
│  │ Null bitmap: ceil(502/8) = 63 bytes                    │ │
│  │ 总计: ~4,094 bytes/行                                   │ │
│  └───────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│  方案B: TOAST 列表式存储                                      │
│  ┌───────────────────────────────────────────────────────┐ │
│  │ 主表 row: [header][ts][pointer→TOAST]                  │ │
│  │            23b   8b   24b = 55 bytes                   │ │
│  └───────────────────────────────────────────────────────┘ │
│                         ↓                                  │
│  ┌───────────────────────────────────────────────────────┐ │
│  │ TOAST 表: [array_header][v1][v2]...[v500]              │ │
│  │             24b         1b 1b     1b                    │ │
│  │                                                       │ │
│  │ 数据: 500 × 8 = 4,000 bytes                            │ │
│  │ Array 头部: 24 bytes                                   │ │
│  │ Varlena 头部: 500 × 1 = 500 bytes (每个元素1字节!)      │ │
│  │ 总计: ~4,524 bytes/行 (+430 bytes 开销)                │ │
│  └───────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────┘
```

#### 每行详细计算对比

| 组成部分 | 方案A | 方案B | 差异 |
|---------|-------|-------|------|
| Tuple Header | 23 bytes | 23 bytes | 0 |
| Timestamp | 8 bytes | 8 bytes | 0 |
| Null Bitmap | 63 bytes (502列) | 1 byte (3列) | **方案B 省 62 bytes** ✅ |
| 数据内容 | 4,000 bytes | 4,000 bytes | 0 |
| **Array 头部** | - | **24 bytes** | **方案B 多 24 bytes** ❌ |
| **Varlena 头部** | - | **500 bytes (500×1)** | **方案B 多 500 bytes** ❌ |
| TOAST 指针 | - | **24 bytes** | **方案B 多 24 bytes** ❌ |
| **总计** | **~4,094 bytes/行** | **~4,580 bytes/行** | **方案B 多 ~486 bytes/行** ❌ |

### 8.7 核心发现

```
┌─────────────────────────────────────────────────────────────┐
│                     TOAST 实验关键结论                        │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ✓ TOAST 已成功触发!                                        │
│    每行 4020 bytes > 2048 bytes 阈值                       │
│                                                             │
│  ✗ 但方案B (TOAST列表) 反而比方案A 大 37.7%!               │
│                                                             │
│  根本原因:                                                  │
│    PostgreSQL 数组每个元素都有 1 byte varlena 头部开销      │
│    500 个元素 × 1 byte = 500 bytes 额外空间                │
│    这个开销远超 null bitmap 节省的空间                      │
│                                                             │
│  TOAST 的真正价值:                                          │
│    不是节省"总存储空间"，而是优化"主表结构"                 │
│    - 主表每行从 ~4094 bytes 缩减到 ~55 bytes               │
│    - 主表每页可存更多行                                     │
│    - 只查询时间戳时 I/O 更快                                │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 九、综合实验总结

### 9.1 三组实验汇总

| 实验 | 列数 | 行数 | 是否触发 TOAST | 结果 |
|------|------|------|----------------|------|
| **原始实验** | 20 | 446 | ❌ 否 (160B < 2KB) | 方案B 小 ~5% (偶然因素) |
| **放大实验** | 20 | 10,000 | ❌ 否 (160B < 2KB) | 方案B 大 ~15% |
| **TOAST 触发实验** | **500** | **1,000** | ✅ 是 (4KB > 2KB) | **方案B 大 ~38%** |

### 9.2 最终结论

| 问题 | 结论 |
|------|------|
| 列合并能节省存储空间吗？ | **❌ 不能**，数组元数据开销更大 |
| 触发 TOAST 后会更好吗？ | **❌ 更差**，Varlena 头部开销显著 |
| 数据量增加能改善吗？ | **❌ 不能**，差异比例基本稳定 |
| TOAST 的真正作用是什么？ | **优化主表 I/O 性能**，非存储密度 |
| 什么场景适合用 TOAST？ | **大文本、JSON、频繁查小字段** |

### 9.3 适用场景建议

| 场景 | 推荐方案 | 原因 |
|------|---------|------|
| **纯数值数据** (< 100列) | **方案A (传统列式)** | 数组开销 > 列合并收益 |
| **纯数值数据** (> 200列) | **方案A (传统列式)** | 即使触发 TOAST 也更占空间 |
| **大文本/BLOB 字段** | **方案B (自动 TOAST)** | 自动压缩 + 外置存储 |
| **需要灵活查询某列** | **方案A (传统列式)** | 可直接索引单列 |
| **批量读写整行数据** | **方案B (TOAST)** | 减少网络往返 |
| **主表查询性能敏感** | **方案B (TOAST)** | 主表更紧凑，扫描更快 |

### 9.4 关键技术要点

```
PostgreSQL 数组类型的隐藏开销:
  1. Array Header: 24 bytes (固定)
  2. Varlena Header: 1 byte per element
  3. 对齐填充: 可能额外增加
  
计算公式:
  数组实际大小 = n×元素大小 + 24 + n + 对齐填充
  
对于 500 个 float8:
  = 500×8 + 24 + 500 + padding
  = 4524+ bytes (比原始数据多 13%!)
  
对于 20 个 float8:
  = 20×8 + 24 + 20 + padding
  = 204+ bytes (比原始数据多 27.5%)
```
